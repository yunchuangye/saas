# 房地产数据采集技术方案

**版本**: 1.0
**作者**: Manus AI
**日期**: 2026-03-13

---

## 1. 概述

本方案旨在为 `gujia.app` 房地产估价 SaaS 平台设计一套合法、高效、可扩展的数据采集系统。该系统将负责从多个来源获取房地产市场的核心数据，包括楼盘信息、挂牌房源、历史成交案例等，为平台的自动估价模型提供数据支持。

## 2. 数据源分析

房地产数据源主要分为三类：政府公开数据、商业数据 API 和公开网络平台。每种数据源都有其优缺点，我们需要结合使用以确保数据的全面性、及时性和准确性。

| 数据源类型 | 主要来源 | 优点 | 缺点 | 采集策略 |
|---|---|---|---|---|
| **政府公开数据** | 各地不动产登记中心、住建委官网、数据开放平台（如杭州、福州）[1][2] | **权威性高**、**合规性强**、数据结构化 | 数据更新频率不一、覆盖城市有限、历史数据不完整、接口不友好 | 优先接入，作为基础数据和验证基准 |
| **商业数据 API** | 中指数据、克而瑞等 [3] | **数据全面**、**结构化**、覆盖范围广、有专门技术支持 | **成本高昂**（通常按城市/年/接口收费）、数据有一定滞后性 | 采购核心城市数据作为爬虫数据的补充和交叉验证 |
| **公开网络平台** | 贝壳找房、链家、安居客、58 同城、房天下等 [4][5] | **数据量大**、**更新快**、覆盖面广、包含丰富的非结构化信息（图片、评论） | **反爬机制强**、数据非结构化、数据质量参差不齐、**法律合规风险高** | 作为主要数据来源，但需采用高级爬虫技术和严格的合规措施 |

## 3. 数据采集技术架构

我们将基于现有的 `saas` 项目后端（Node.js + Express）进行扩展，集成一个强大的数据采集模块。该模块将采用**分布式主从架构**，由一个中心化的**调度器 (Scheduler)** 和多个**执行器 (Worker)** 组成。

```mermaid
graph TD
    subgraph 前端管理界面
        A[任务配置] --> B{后端 API}
        C[状态监控] --> B
        D[数据预览] --> B
    end

    subgraph 后端数据采集模块
        B --> E[调度器 Scheduler]
        E --> F{任务队列 (Redis)}
        F --> G1[执行器 Worker 1]
        F --> G2[执行器 Worker 2]
        F --> G3[执行器 Worker ...]
    end

    subgraph 执行器 Worker
        G1 --> H[爬虫引擎 (Playwright)]
        H --> I{代理池}
        I --> J[目标网站]
        H --> K[数据解析/清洗]
        K --> L[数据暂存 (Redis)]
        L --> M[数据入库 (MySQL)]
    end

    J -- 反爬对抗 --> I
    M -- 写入 --> N[gujia 数据库]
```

### 3.1. 核心组件说明

- **调度器 (Scheduler)**: 负责从前端接收采集任务，将任务分解（如按城市、区域、楼盘）后放入任务队列。同时负责监控 Worker 状态和任务进度。
- **任务队列 (Task Queue)**: 使用 Redis 的 List 或 Stream 实现，存储待采集的 URL 或任务单元，实现任务的持久化和分布式调度。
- **执行器 (Worker)**: 无状态的爬虫进程，从任务队列中获取任务并执行。每个 Worker 独立运行，可水平扩展。
- **爬虫引擎 (Crawling Engine)**: 核心采集工具，选用 **Playwright**。相比传统的 `requests` + `BeautifulSoup`，Playwright 是一个无头浏览器框架，能有效应对现代网站普遍采用的 JavaScript 动态渲染和复杂的前端反爬技术。
- **代理池 (Proxy Pool)**: 为避免 IP 被封禁，所有爬虫请求都将通过代理 IP 发出。我们将集成一个动态代理池服务，该服务能提供大量高匿名的住宅 IP，并自动轮换失效 IP。
- **数据解析与清洗**: 爬取到的原始 HTML 数据将经过解析，提取出目标字段（如价格、面积、户型、楼层等）。同时进行数据清洗，统一单位（如“万元”转为“元”），格式化日期，处理异常值。
- **数据入库**: 清洗后的结构化数据将暂存到 Redis，然后批量异步写入 MySQL 数据库的对应表中（`cases`, `estates` 等）。

### 3.2. 反爬虫策略

| 策略 | 实现方式 |
|---|---|
| **动态 IP 代理** | 集成第三方商业代理池服务，为每个请求或会话分配不同的 IP 地址。 |
| **模拟真人行为** | - 使用 Playwright 模拟完整的浏览器环境（User-Agent, cookies, headers）。<br>- 在操作之间插入随机延迟。<br>- 模拟鼠标移动、滚动页面等交互动作。 |
| **验证码处理** | - 优先选择不易触发验证码的采集路径。<br>- 对接第三方打码平台（如 2Captcha）API，自动识别和输入图形/滑块验证码。 |
| **动态渲染处理** | Playwright 原生支持执行 JavaScript，可等待动态内容加载完成后再抓取。 |
| **请求指纹对抗** | Playwright 可配置浏览器指纹（如 Canvas、WebGL、字体），降低被识别为机器人的概率。 |

## 4. 数据采集流程

1. **任务创建**: 用户在前端“数据采集”页面选择目标城市、数据类型（如“二手房成交”），设定采集深度（如最近 100 页），创建采集任务。
2. **任务分发**: 后端调度器接收任务，生成初始 URL 列表，并将其推送到 Redis 任务队列。
3. **数据抓取**: 多个 Worker 进程从队列中消费 URL，使用 Playwright 打开页面，执行抓取逻辑。
4. **数据解析**: 从页面 HTML 中提取所需字段，进行初步清洗和格式化。
5. **数据存储**: 将解析后的数据存入 MySQL 数据库。
6. **循环与深度抓取**: 如果在页面中发现新的链接（如“下一页”、楼盘详情页），则将新链接重新推送到任务队列，实现深度抓取。
7. **任务完成与监控**: 当任务队列为空时，任务完成。整个过程中的任务状态、成功/失败数量、数据预览等信息会实时更新到前端页面。

## 5. 合规性与风险管理

数据采集，特别是网络爬虫，存在法律和商业风险。我们将采取以下措施确保合规：

- **遵守 `robots.txt`**: 在采集前检查目标网站的 `robots.txt` 文件，尊重网站的爬取规则。
- **降低采集频率**: 设置合理的请求间隔，避免对目标服务器造成过大压力。
- **明确数据用途**: 采集的数据仅用于平台内部的估价模型分析，不直接对外展示或二次分发原始数据。
- **数据脱敏**: 对于可能涉及个人信息的数据（如业主联系方式），在采集和存储过程中进行脱敏处理。
- **优先使用官方渠道**: 优先从政府公开平台和付费商业 API 获取数据，将网络爬虫作为补充手段。
